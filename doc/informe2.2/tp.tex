\section{Enunciado}

\vspace{1cm}
\Large \textbf{Objetivo}\\
\normalsize

\begin{itemize}
\item Taggear tests (un string, ejemplo: FAST, SLOW, DB, INTERNET, etc)
\item Poder agregar más de un tag a un test.
\item Permitir desactivar un test con SKIP.
\item Poder ejecutar test pertenecientes a un tag particular.
\item Poder ejecutar test pertenecientes a varios tags.
\item Poder ejecutar test pertenecientes a varios tags y/o que su nombre de test case
coincida con una expression regular y/o que su nombre de test suite coincida con una
expresión regular.
\item Tomar el tiempo a todos los test para poder agregarlos en reportes.
\item Reporte textual por línea de comando. (Utilizar formato de la entrega anterior).
\item Que el reporte textual por línea de comando anterior sea progresivo en real time (a medida que vaya teniendo resultados los muestra).
\item Implementar un reporte xml siguiendo el siguiente XSD: https://svn.jenkinsci.
org/trunk/hudson/dtkit/dtkitformat/dtkitjunitmodel/src/main/resources/com/thalesgroup/dtkit/junit/model/xsd/junit4.xsd\\
\end{itemize}

\Large \textbf{Formato de salida del Reporte XML}\\\
\normalsize

\noindent Ver:\\
https://svn.jenkinsci.org/trunk/hudson/dtkit/dtkitformat/dtkitjunitmodel/src/main/resources/com/thalesgroup/dtkit/junit/model/xsd/junit4.xsd\\\\

\Large \textbf{Restricciones}
\normalsize

\begin{itemize}
\item Trabajo Práctico grupal implementado en java o C\#
\item Se deben utilizar las mismas herramientas que en el TP0 (git + maven + junit4 / git
+ VS 2012 + MS Test o NUnit).
\item Todas las clases del sistema deben estar justificadas.
\item Se debe modelar utilizando un modelo de dominio, y no usando herramientas
tecnológicas como reflection, annotations, etc.

misma.

de su conveniencia por sobre otras opciones.
\item Se debe tener una cobertura completa del código por tests
\item Se pide ademas de tener los test unitarios junit/nunit.
\end{itemize}

\section{Justificaciones}

\begin{itemize}
\item \textbf{Herencia}:
	Abstract Test, AbstractUnitTest, TestSuit\\
	\underline{Motivación}:\\ Reducir la duplicidad de código. Los AbstractUnitTest y los TestSuit tienen caracteristicas comunes a ambos, las cuales agrupamos en la clase AbstractTest. En principio solo encontramos la necesidad de un nombre como caracteristica comun, pero podría haber mas, entonces esto ayuda también a encapsular el cambio.\\

	

\item \textbf{Patrones utilizados:}

	\underline{Builder}:\\
		\underline{Utilizado} en: TestResult.java, clase ?Builder?\\
		\underline{Motivación}: Se requería construir un TestResult (resultado de un test), como resultado de distintos eventos que modificaban el estado del mismo. El Builder permite hacer esto de una forma clara.\\

	\underline{Composite}:\\
		\underline{Utilizado en}: Test, TestSuite\\
		\underline{Motivación}: La forma en que se organizan los test con la estructura de un arbol presentan una oportunidad de utilizar este patrón beneficianos de sus ventajas y sin tener que lidiar con sus desventajas (la interfaz de los componentes es válida tanto para elementos hoja como para elementos no hoja).\\

	\underline{Visitor}:\\
		\underline{Utilizado} en: TestResultCollector\\
		\underline{Motivación}: Simplifica la recolección de resultados en la jerarquia de tests.
		
	\underline{Command}:\\
		\underline{Utilizado} en: TestRunner\\
		\underline{Motivación}: Simplificación de interfaz de runner.

\item \textbf{Clases}:

	Las justificaciones pueden encontrarse en el codigo como comentario en cada clase, aquí se expone un resumen de las mismas.\\

	\textbf{Interfaces:}\\\\
		Test:\\
			\indent Interfaz genérica para un test.\\\\
		TestResult:\\
			\indent Segregacion de interfaz.\\\\
		TestResultCollector:\\
			\indent Separar implementacion del colector de resultados.\\\\
		TestRunner:\\
			\indent Segregacion de interfaz.\\
		TestReporter:\\
			\indent Interfaz de Test Reporter

		Assert1:\\
			\indent De esta forma los asserts que tengan el tipo de firma de Assert1 (recibiendo solo 1 parametro) solo deben redefinir el metodo apply para poder ser utilizados.\\\\
		Assert2:\\
			\indent Igual Assert1.\\\\
		Context:\\
			\indent Persistencia a travez de la corrida de los tests, de los elementos creados en los Setup.\\\\
	\textbf{Clases Concretas:}\\\\
		AssertXXXXX:\\
			\indent Implementan los distintos tipos de verificación.\\\\
		Asserts:\\
			\indent Permite utilizar los asserts sin tener que crear instancias de los mismos.\\
		AbstractUnitTest:\\
			\indent Implementacion para los tests unitarios.\\\\
		TestResultCollectorImpl:\\
			\indent Implementacion del colector de resultados. Para poder obtener todos los resultados en el mismo lugar, en el caso de estar corriendo una estructura tipo arbol de tests.\\\\
		TestResultImpl:\\
			\indent Implementación de Resultado de tests, tiene cierta información que se cree de utilidad para quien ha corrido los tests.\\\\
		TestRunnerImpl:\\
			\indent Implementación del TestRunner, permite correr un test o suite de tests y obtener un resumen sobre los resultados.\\\\
		TestSuite:\\
			\indent Contiene una colección de tests, permite organizar y correr tales tests, y tambien permite correr otros TestSuites, lo que otorga la posibilidad de correr los tests con una estructura del tipo arbol.\\\\
		ContextImpl:\\
			\indent Implementación de Persistencia de contexto.\\
		TestConsoleReporter:\\
			\indent Implementación del reporte progresivo de resultados de tests
		XmlMapper:\\
			\indent Mapeador de las clases a XML.
\end{itemize}
